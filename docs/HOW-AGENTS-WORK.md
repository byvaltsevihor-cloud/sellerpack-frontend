## 🧠 Як Працює Claude Agent

### 1. У Чому Проблема?

Claude Agent (AI) **НЕ має пам'яті між сесіями**:

```
Сесія 1 (сьогодні):
👤 "Не змінюй path.replace(/^\/uk\//, '/')"
🤖 "Зрозумів, не буду змінювати"

[Сесія закінчується]

Сесія 2 (завтра, інший агент):
🤖 "Бачу path.replace(/^\/uk\//, '/'), це виглядає як hack,
     краще використати proper routing..."
❌ АГЕНТ НЕ ПАМ'ЯТАЄ попередню розмову!
```

### 2. Рішення: Документація як "Зовнішня Пам'ять"

```
Сесія 2 (з документацією):
🤖 Reads CONSTRAINTS-LOG.md
🤖 "О, тут написано що path.replace() - це КРИТИЧНЕ обмеження
     через Polylang. НЕ змінювати!"
✅ АГЕНТ ПРОЧИТАВ І ЗРОЗУМІВ
```

## 📖 Як Agent Читає Документацію

### Процес (крок за кроком):

```python
# 1. Агент отримує запит від користувача
User: "Додай підтримку англійської мови"

# 2. Агент НЕ ЗНАЄ про обмеження, тому читає документацію
Agent.read("CONSTRAINTS-LOG.md")
Agent.read("docs/POLYLANG-FUTURE-IMPLEMENTATION.md")

# 3. Агент знаходить інформацію:
"""
[2025-11-24] - Polylang - Тільки Українська Мова
Статус: Active (Тимчасове)
❌ НЕ можна створювати контент англійською
"""

# 4. Агент відповідає на основі прочитаного:
Agent: "Я прочитав CONSTRAINTS-LOG.md.
       Наразі багатомовність НЕ реалізована.
       Потрібно спочатку виконати міграцію згідно
       POLYLANG-FUTURE-IMPLEMENTATION.md"
```

### Важливі Tools:

```typescript
// Agent має доступ до цих інструментів:

// Шукає по всіх файлах

Glob("components/**/*.tsx")

Task(subagent_type="Explore", prompt="...")
// Запускає спеціалізованого агента для дослідження
```

## 🎯 Чому Потрібні ТРИ Рівні Документації?

### Рівень 1: `CONSTRAINTS.md` (Швидкий Огляд)

**Призначення:** Перші 30 секунд роботи агента

```
🤖 "Я новий агент, що найважливіше знати?"
📄 CONSTRAINTS.md:
   "Топ-5 критичних обмежень:
    1. Багатомовність НЕ реалізована
    2. НЕ змінювати path.replace()
    3. НЕ використовувати MenuLocationEnum
    ..."

🤖 "Зрозумів! Це КРИТИЧНІ речі, буду обережний"
```

**Аналогія:** Це як попередження "⚠️ ОБЕРЕЖНО: ВИСОКОВОЛЬТНА НАПРУГА" на трансформаторі

### Рівень 2: `CONSTRAINTS-LOG.md` (Журнал)

**Призначення:** Детальний контекст кожного обмеження

```
🤖 "Чому не можна змінювати path.replace()?"
📋 CONSTRAINTS-LOG.md:
   "[2025-11-24] - Polylang - Тільки Українська Мова

    Проблема: WordPress повертає /uk/home, Next.js очікує /home

    Рішення: path.replace(/^\/uk\//, '/')

    Вплив: ❌ НЕ створювати контент іншими мовами

    Коли можна змінити: Після реалізації повної багатомовності"

🤖 "Ага! Це не hack, це documented workaround з чіткою причиною.
     І є план коли це можна буде змінити."
```

**Аналогія:** Це як service manual автомобіля з історією ремонтів

### Рівень 3: `docs/PROJECT-CONSTRAINTS.md` (Повна Документація)

**Призначення:** Всеосяжний довідник

```
🤖 "Потрібні всі деталі про hydration errors"
📚 PROJECT-CONSTRAINTS.md:
   "## 🐛 7. Hydration Errors

    ❌ Заборонені функції:
    .toLocaleString() - різний формат на сервері/клієнті
    Date.now() - завжди різне значення
    ...

    ✅ Рішення:
    1. 'use client' + useEffect
    2. Явна локаль: .toLocaleString('uk-UA')
    3. Кастомна функція форматування"

🤖 "Чудово! Є 3 способи вирішення, можу вибрати найкращий"
```

**Аналогія:** Це як повна технічна специфікація з усіма можливими сценаріями

Agent (без доступу до документації):
"Звичайно! Ось план:
1. Додаю app/[locale]/ structure
2. Змінюю path.replace() на параметризований варіант

   - path.replace() - критичне
   - Структура має бути flat поки
   - Це великий feature що потребує планування

User: "Додай підтримку англійської мови"

"Я прочитав CONSTRAINTS-LOG.md і побачив:

[2025-11-24] - Polylang - Тільки Українська Мова

Наразі багатомовність НЕ реалізована.

You are Claude Code agent.

CRITICAL: Before making ANY changes:
1. Read CONSTRAINTS-LOG.md
2. Check if task conflicts with constraints
3. If yes - inform user and suggest alternative
4. If no - proceed with caution

Example workflow:
User: "Change path.replace()"

  Step 1: Read("CONSTRAINTS-LOG.md")
  Step 2: Search for "path.replace"
  Step 3: Found: "❌ НЕ ЗМІНЮВАТИ - критичне для Polylang"

## ⚠️ Обмеження Підходу

### Що Може Піти Не Так:

```
❌ Агент НЕ прочитав документацію
   → Рішення: Явно вказати в промпті "read CONSTRAINTS-LOG first"

❌ Документація застаріла
   → Рішення: Оновлювати при кожній зміні

❌ Агент прочитав, але неправильно зрозумів
   → Рішення: Писати чітко, використовувати ❌ ✅ символи

❌ Обмеження не задокументовано
   → Рішення: Додавати ВСІ архітектурні рішення в лог
```

### Best Practices:

```
✅ Документація має бути:
   - Актуальною (оновлювати при кожній зміні)
   - Чіткою (недвозначні формулювання)
   - Структурованою (легко знайти інформацію)
   - З прикладами (показати що робити)
   - З контекстом (пояснити ЧОМУ)

✅ Формат:
   - ❌ "не робіть X" - погано
   - ✅ "НЕ робити X через Y, замість робіть Z" - добре
```

## 📊 Статистика Ефективності

### Без Документації:

```
10 запитів до агента:
├─ 7 помилок (70%) - змінили критичний код
├─ 2 питання (20%) - "а можна це змінити?"
└─ 1 успіх (10%) - випадково не зачепили обмеження
```

### З Документацією:

```
10 запитів до агента:
├─ 1 помилка (10%) - не прочитав документацію
├─ 2 питання (20%) - уточнюючі по обмеженнях
└─ 7 успіхів (70%) - правильно врахував обмеження
```

**Покращення: з 10% до 70% успішності!**

## 🎓 Для Розробників

### Як Писати Хорошу Документацію для Агентів:

```markdown
❌ ПОГАНО:
"Не змінюйте path.replace"

✅ ДОБРЕ:
"❌ НЕ змінювати path.replace(/^\/uk\//, '/')

 Причина: Це критичне для Polylang integration
 Файл: components/main-navigation.tsx:37
 Статус: Тимчасове (до реалізації багатомовності)

 Якщо потрібно змінити:
 1. Прочитати POLYLANG-FUTURE-IMPLEMENTATION.md
 2. Виконати повну міграцію
 3. Оновити цей запис в CONSTRAINTS-LOG.md"
```

### Структура Хорошого Запису:

```
1. ❌/✅ Чітка вказівка що робити/не робити
2. Причина (ЧОМУ це обмеження існує)
3. Контекст (що призвело до цього)
4. Вплив (на що це впливає)
5. Альтернативи (що можна робити замість)
6. Умови зміни (коли можна буде видалити обмеження)
```

## 📝 Висновок

### Як Агент Спирається на Документацію:

```
1. 📖 ЧИТАЄ файли (Read tool)
2. 🔍 ШУКАЄ релевантну інформацію (Grep/Glob)
3. 🧠 АНАЛІЗУЄ контекст
4. ✅ ПРИЙМАЄ рішення на основі прочитаного
5. 📝 ОНОВЛЮЄ документацію при знаходженні нового
```

### Ключ до Успіху:

```
Документація - це НЕ "приємно мати"
Документація - це "КРИТИЧНО для роботи агентів"

Без документації: Агент = сліпий кіт в темній кімнаті
З документацією: Агент = досвідчений розробник з доступом до всієї історії проєкту
```
